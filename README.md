# A1-1 프롬프트 관리 프로그램

자주 쓰는 AI 프롬프트를 저장하고 관리하는 Python 콘솔 프로그램입니다.
카테고리 분류·검색·즐겨찾기로 필요한 프롬프트를 빠르게 찾을 수 있습니다.

저장소: https://github.com/reinvent21c-oss/A1-1

## 프로젝트 소개

AI를 사용하다 보면 좋은 프롬프트를 매번 다시 찾게 됩니다.
메모장·노션·메신저에 흩어진 프롬프트를 한곳에 모으고,
카테고리와 즐겨찾기로 관리하기 위해 만들었습니다.

기본 데이터로는 이전 미션에서 **실제로 작성해 사용한 프롬프트 3개**를 등록해 두었습니다.

## 카테고리 종류

프롬프트는 다음 6가지로 분류합니다.

- 텍스트 생성
- 이미지 생성
- 영상 생성
- 페르소나
- 자동화
- 기타

카테고리 목록은 전역 변수 `categories` 한 곳에서 관리합니다.
카테고리 체계가 바뀌면 이 리스트만 수정하면 되도록 설계했습니다.
프롬프트 추가 시에는 목록에 없는 카테고리도 직접 입력할 수 있습니다.

## 데이터 구조

프롬프트 하나는 5개 키를 가진 **딕셔너리**이고, 이들을 **리스트**에 담아 관리합니다.

```python
prompts = [
    {
        "title": "킥오프 회의록 정리 (웹에이전시 PM)",
        "content": "너는 웹에이전시 프로젝트 담당자(PM)를 보조하는...",
        "category": "페르소나",
        "favorite": False,
        "views": 0
    },
    ...
]
```

**딕셔너리를 쓴 이유**
프롬프트 하나가 가진 5개 속성을 이름으로 꺼낼 수 있습니다.
`p["category"]`처럼 의미가 드러나서, 순서에 의존하는 구조보다 읽고 고치기 쉽습니다.

**리스트를 쓴 이유**
순서가 유지되고 번호로 접근할 수 있어, 메뉴에서 "몇 번 프롬프트"를 고르는 방식과 그대로 맞습니다.
추가·삭제 시 나머지 항목이 자동으로 밀리고 당겨집니다.

**이 구조의 단점**
- 검색이 전체 순회 방식이라 개수가 늘어나면 느려집니다. 항목이 많아지면 제목을 키로 하는 딕셔너리나 데이터베이스가 유리합니다.
- 화면 번호가 리스트 위치에 묶여 있어, 삭제하면 그 아래 항목의 번호가 전부 바뀝니다. 고유 ID를 따로 두면 해결되지만 지금 규모에서는 과한 설계라고 판단했습니다.
- 같은 제목의 프롬프트가 여러 개 생길 수 있습니다. 현재는 화면 번호로 구분하며, 필요해지면 추가 시 제목 중복을 확인하는 방식으로 보완할 수 있습니다.

## 데이터 영속화 (보너스 1-a)

프로그램을 종료해도 데이터가 남도록 `prompts.json` 파일에 저장합니다.

**파일 형식으로 JSON을 고른 이유**
파이썬의 리스트·딕셔너리 구조를 그대로 담을 수 있어 변환 과정이 필요 없습니다.
표준 라이브러리 `json`만으로 처리되어 외부 라이브러리를 쓰지 않아도 됩니다.
텍스트 파일이라 사람이 직접 열어 확인할 수 있는 것도 장점입니다.

**저장 시점 — 바뀔 때마다**
종료 시 한 번 저장하면 강제 종료나 오류 발생 시 그 사이 변경이 모두 사라집니다.
그래서 데이터가 바뀌는 지점마다 `save_prompts()`를 호출합니다.
추가, 즐겨찾기 토글, 수정, 삭제, 상세보기(조회수 증가) 다섯 곳입니다.
검색·목록·카테고리 조회는 읽기만 하므로 저장하지 않습니다.

**불러오기와 실패 처리**
시작 시 `prompts = load_prompts(prompts)`로 파일을 읽습니다.
저장과 불러오기는 한 쌍이며, 불러오기가 없으면 파일에는 기록되지만 매번 기본 데이터로 시작하게 됩니다.

| 상황 | 처리 |
|:---|:---|
| 파일 없음 (첫 실행) | 조용히 기본 데이터로 시작 — 오류가 아니라 정상 상태 |
| 파일이 깨짐 | 경고 후 기본 데이터로 시작 — 프로그램을 중단하지 않음 |

깨진 파일로 시작한 뒤 데이터가 한 번이라도 바뀌면 정상 JSON으로 덮어써져 파일이 복구됩니다.

한글이 `\uXXXX` 형태로 저장되지 않도록 `ensure_ascii=False`,
사람이 읽을 수 있도록 `indent=2`를 지정했습니다.

## 보너스 과제 구현 내역

| 과제 | 구현 |
|:---|:---|
| 1-a 프롬프트 영속화 | `prompts.json` 저장·불러오기 (위 항목 참고) |
| 1-b Markdown 내보내기 | 메뉴 11번. `exports/` 폴더에 카테고리별 `.md` 생성 |
| 2-a 수정/삭제 | 메뉴 8·9번 |
| 2-b 사용 횟수 기록 | 상세보기(6번) 시 `views` +1 후 저장 |
| 2-c 조회수 기준 정렬 | 메뉴 10번. `sorted()`의 `key`에 `get_views` 함수 전달 |

**Markdown 내보내기 설계**
미션의 "카테고리별 Markdown 파일"을 카테고리당 파일 하나로 해석했습니다.
미리 정의된 6개 카테고리를 도는 대신 **실제 데이터에 존재하는 카테고리만** 수집합니다.
사용자가 직접 입력한 카테고리도 누락되지 않고, 비어 있는 카테고리 파일도 생기지 않습니다.

**조회수 정렬에서 이름 있는 함수를 쓴 이유**
`sorted(prompts, key=get_views, reverse=True)` 형태로, 정렬 기준을 별도 함수로 분리했습니다.
기준이 이름으로 드러나 읽기 쉽고, 다른 곳에서도 재사용할 수 있습니다.

**기존 데이터와의 호환**
`views` 항목은 나중에 추가된 필드입니다.
그래서 읽을 때 `p.get("views", 0)`을 써서, 항목이 없는 옛 데이터도 0으로 처리되도록 했습니다.

## 코드 구조

모든 코드를 한 곳에 몰아넣지 않고 기능별로 함수를 분리했습니다.

| 구분 | 함수 |
|:---|:---|
| 파일 입출력 | `load_prompts()`, `save_prompts()` |
| 입력 보조 | `input_required()` |
| 조회 | `show_menu()`, `show_all()`, `show_by_category()`, `show_detail()`, `show_favorites()`, `show_top_views()` |
| 변경 | `add_prompt()`, `edit_prompt()`, `delete_prompt()`, `toggle_favorite()` |
| 검색·정렬 | `search_prompt()`, `get_views()` |
| 내보내기 | `export_markdown()` |
| 실행 흐름 | `main()` |

**입력 검증 위치**
빈 값 검증은 `input_required()` 한 곳에 모았습니다. 값이 들어올 때까지 반복해서 묻습니다.
번호 범위 검증은 번호를 받는 각 함수 안에서 처리합니다.
`isdigit()`으로 숫자 여부를, 이어서 1 이상 개수 이하인지를 확인하고, 벗어나면 안내 후 `return`으로 메뉴에 돌아갑니다.

**메뉴 반복에 `while`을 쓴 이유**
기능을 몇 번 사용할지 미리 알 수 없기 때문입니다.
`while True`로 계속 돌리고, 사용자가 `0`을 선택했을 때만 `break`로 빠져나옵니다.

## 커밋 전략

기능 단위로 커밋하고, 커밋 메시지 앞에 종류를 표시했습니다.

| 접두어 | 의미 | 실제 예시 |
|:---|:---|:---|
| `feat` | 기능 추가 | `feat: JSON 파일로 프롬프트 영속화 (불러오기/저장)` |
| `fix` | 버그 수정 | `fix: 즐겨찾기 관리에 번호 입력 검증 추가` |
| `chore` | 설정 변경 | `chore: 실행 중 생성되는 prompts.json 무시 규칙 추가` |
| `docs` | 문서 작업 | `docs: 데이터 구조 및 브랜치 전략 설명 추가` |

**나누는 기준은 "성격이 다른가"입니다.**
예를 들어 영속화 작업은 `main.py`의 기능 추가(`feat`)와 `.gitignore` 규칙 추가(`chore`)로 나눴습니다.
하는 일이 다르므로 되돌릴 때도 따로 되돌릴 수 있어야 하기 때문입니다.

반대로 **하나의 기능을 억지로 쪼개지는 않았습니다.**
메뉴에는 항목이 있는데 함수는 아직 없는 상태처럼, 실행되지 않는 중간 커밋이 생기기 때문입니다.

전체 커밋 이력은 `screenshots/git로그.png`에서 확인할 수 있습니다.

## 브랜치 전략

- `main` — 배포 가능한 안정 버전
- `feature/json-persist` — 보너스 과제(영속화·내보내기·수정·삭제·조회수)를 개발한 브랜치

**브랜치를 나눈 이유**
보너스 과제는 필수 기능이 모두 완성된 뒤 얹는 작업입니다.
작업 도중 문제가 생겨도 `main`은 언제나 정상 동작하는 상태로 남겨두기 위해 분리했습니다.

**병합 시점**
보너스 1과 2를 모두 완성하고 전체 기능 테스트를 통과한 뒤 한 번에 병합했습니다.
기능이 하나 끝날 때마다 병합하지 않은 것은, 메뉴 번호 체계처럼 여러 기능에 걸친 변경이 중간에 반쯤 반영되는 상태를 피하기 위해서입니다.

**`--no-ff` 옵션을 쓴 이유**

```bash
git merge --no-ff --no-edit feature/json-persist
```

`main`이 분기 이후 움직이지 않은 경우, 기본 병합은 fast-forward로 처리됩니다.
새 커밋 없이 `main`이 가리키는 위치만 앞으로 옮기는 방식입니다.
결과 파일은 같지만 이력이 일직선이 되어, 브랜치에서 작업했다는 사실이 남지 않습니다.

`--no-ff`는 부모가 둘인 병합 커밋을 만들어 분기와 병합 구조를 이력에 남깁니다.
`--no-edit`은 Git 기본 메시지를 그대로 사용해 `Merge branch 'feature/json-persist'`처럼
브랜치 이름이 커밋 메시지에 기록되게 합니다.

이 프로젝트 초기에 `feature/favorite-toggle` 브랜치를 fast-forward로 병합해
분기 기록이 남지 않은 경험이 있어, 이후 병합에는 `--no-ff`를 적용했습니다.

## 병합 충돌 대응 절차

충돌은 같은 파일의 같은 부분을 양쪽 브랜치에서 다르게 고쳤을 때 발생합니다.
다음 순서로 처리합니다.

**1단계 — 원인 파악**

```bash
git status
```

`both modified`로 표시된 파일이 충돌 지점입니다.
파일을 열면 `<<<<<<<`, `=======`, `>>>>>>>` 표시로 양쪽 내용이 구분되어 있습니다.

**2단계 — 해결**
표시 사이의 두 버전을 비교해 최종 내용을 직접 작성하고, 표시 기호를 모두 지웁니다.
어느 한쪽을 고르거나, 양쪽을 합치거나, 새로 쓰는 것 모두 가능합니다.
판단 기준은 "어느 쪽이 최신인가"가 아니라 "합친 결과가 정상 동작하는가"입니다.

**3단계 — 검증**

```bash
python main.py
```

먼저 프로그램이 정상 실행되는지 확인합니다. 표시 기호가 남아 있으면 문법 오류가 납니다.
동작을 확인한 뒤 커밋합니다.

```bash
git add <파일명>
git commit
```

충돌 해결 커밋은 무엇을 왜 그렇게 합쳤는지 메시지에 남깁니다.

**충돌을 줄이는 방법**
브랜치 수명을 짧게 가져가고, 한 브랜치에서 여러 파일을 광범위하게 고치지 않는 것이 예방책입니다.

## Git으로 관리하지 않는 파일

`.gitignore`에 다음을 등록했습니다.

| 대상 | 이유 |
|:---|:---|
| `prompts.json` | 실행 중 생성되는 사용자 데이터. 기본 데이터가 코드에 있어 clone 후에도 정상 동작 |
| `exports/` | 내보내기 결과물. 언제든 다시 생성 가능 |
| `__pycache__/` | 파이썬 실행 시 자동 생성되는 캐시 |
| `.DS_Store`, `.vscode/` | macOS·편집기가 만드는 개인 환경 파일 |

소스가 아니라 **결과물**이라는 공통점이 있습니다.

## 배운 점

- **저장과 불러오기는 한 쌍이다** — 저장 코드만 넣고 불러오기를 빠뜨리면 파일에는 기록되지만 프로그램은 매번 기본 데이터로 시작합니다. 불러오는 줄을 뺀 상태와 넣은 상태를 직접 비교해보고 확인했습니다.
- **읽기 실패는 중단이 아니라 대체로 처리한다** — 파일이 깨져도 프로그램이 죽지 않고 기본 데이터로 계속 동작하게 만드는 편이 낫습니다.
- **fast-forward는 오류가 아니라 Git의 정상 동작이다** — 다만 이력에 분기가 남지 않으므로, 브랜치 작업을 기록으로 남기려면 `--no-ff`가 필요합니다.
- **함수 밖에서 대입하면 `global` 선언이 필요 없다** — 함수 안에서 전역 변수에 새 값을 대입하려면 `global` 선언이 필요합니다. 불러오기 대입을 함수 바깥에 두어 이 문제를 피했습니다.
- **필드를 추가할 때는 옛 데이터를 고려한다** — `views` 추가 후 `.get("views", 0)`으로 읽어, 해당 항목이 없는 기존 저장 파일도 문제없이 동작하게 했습니다.
